# ScopeLock ‚Äî Phase 1 Report

## Repository stack found

This directory previously contained a *different* hackathon project called
"Runway" (an autonomous cash-flow guardian): a plain Node.js/Express server
(`server.js`) with a vanilla JS/HTML/CSS frontend, its own ad-hoc Xero client,
and `@anthropic-ai/sdk` for AI negotiation drafts ‚Äî no TypeScript, no Next.js,
no Tailwind, no database. That app's purpose (cash-flow forecasting) was
unrelated to ScopeLock (scope-creep protection), and its stack didn't match
this spec's requirements (Next.js App Router, TypeScript, Tailwind, Prisma).

Per explicit user confirmation, the old app was removed and ScopeLock was
scaffolded fresh in this same directory rather than adapted, since adapting
would have meant replacing the server, frontend, and Xero layer wholesale
anyway.

## Database changes

New SQLite database (`dev.db`, gitignored) via Prisma, with one model:

```prisma
model XeroConnection {
  id                String   @id @default(cuid())
  tenantId          String
  tenantName        String
  encryptedTokenSet String
  createdAt         DateTime @default(now())
  updatedAt         DateTime @updatedAt
}
```

Single-user hackathon MVP: `token-store.ts` always reads/writes a fixed
`id: "singleton"` row via upsert/delete ‚Äî at most one connection ever exists.
`encryptedTokenSet` stores `{ tokenSet, connectionId }` as one AES-256-GCM
encrypted JSON blob (the Xero "connection id" needed for `disconnect()` isn't
a token, but keeping it alongside the token set inside the single encrypted
column avoided widening the schema beyond what the spec asked for).

**Post-hackathon**: switch `datasource.provider` in `prisma/schema.prisma`
from `sqlite` to `postgresql`, point `DATABASE_URL` at a real Postgres
instance, install a Postgres Prisma driver adapter, and re-run
`prisma migrate dev`. (Prisma 7 requires an explicit driver adapter passed to
`new PrismaClient({ adapter })` ‚Äî see `src/lib/prisma.ts`.)

## Routes implemented

| Route | Method | Purpose |
| --- | --- | --- |
| `/api/xero/connect` | GET | Redirects to Xero consent, sets a signed httpOnly `xero_oauth_state` cookie |
| `/api/xero/callback` | GET | Validates state, exchanges code for tokens, saves connection, redirects to `/xero` |
| `/api/xero/disconnect` | POST | Best-effort Xero-side revoke, always clears local connection |
| `/api/xero/status` | GET | Returns only `{connected, tenantId?, tenantName?, tokenExpiresAt?, permissionsNeedUpdate?}` ‚Äî never tokens |
| `/api/xero/quotes` | GET | Normalised quotes, optional `?status=` filter |
| `/api/xero/contacts` | GET | Normalised contacts |
| `/api/xero/items` | GET | Normalised items |
| `/api/xero/test-connection` | POST | Runs the 3 read checks + a scope-only "draft quote writable" check (never writes) |
| `/api/xero/test-quote` | POST | Creates exactly one DRAFT quote; requires `{contactId, confirmed: true}` |

## Security decisions

- All Xero SDK calls live in `src/lib/xero/*`, imported only from Route
  Handlers (server-only execution context). Every file in that directory
  (and `crypto.ts`, `prisma.ts`) starts with `import "server-only"`, so an
  accidental client-side import fails the build instead of silently bundling
  secrets.
- The stored token set is encrypted with AES-256-GCM (`src/lib/crypto.ts`)
  before it touches disk; the key comes from `TOKEN_ENCRYPTION_KEY` and is
  validated to decode to exactly 32 bytes.
- No API route ever returns `access_token`/`refresh_token` ‚Äî `/api/xero/status`
  returns a fixed, hand-picked JSON shape, and errors are converted through
  `toXeroAppError()` (`src/lib/xero/errors.ts`) which strips raw Xero response
  bodies down to a safe `{error, code, actionHint}`.
- OAuth `state` is a random 16-byte hex value stored in an httpOnly,
  `sameSite=lax` cookie and compared against Xero's callback `state` param ‚Äî
  standard CSRF protection for the authorization-code flow.
- The "draft quote writable" integration check never creates a quote ‚Äî it
  only checks whether `accounting.invoices` (the scope Quotes now lives
  under) was granted, and says so explicitly in its result message.
- The test-draft-quote panel requires an explicit checkbox
  ("I confirm I am using a Xero Demo Company or test organisation") before
  the button becomes clickable, and the button/query disables itself while
  the request is in flight to avoid duplicate quotes from a double-click.

## Live Xero tests that passed

A real Xero app was registered and connected against the developer's own
Xero organisation. Confirmed working end to end, against the live Xero API:

- Full OAuth round trip: `/api/xero/connect` ‚Üí Xero login/consent ‚Üí
  `/api/xero/callback` ‚Üí encrypted token saved ‚Üí `GET /api/xero/status`
  returns `{"connected": true, "tenantName": "Yariel", "tokenExpiresAt": ...,
  "permissionsNeedUpdate": false}`.
- `GET /api/xero/contacts` ‚Äî **Passed** (real contacts returned, normalised).
- `GET /api/xero/items` ‚Äî **Passed** (real items returned, normalised).
- "Run connection tests" against the live connection: Contacts and Items
  read checks passed; Quotes read and the draft-quote-writable scope check
  correctly reported **Failed** with an accurate reason (see below) ‚Äî this is
  expected, not a bug.
- `GET /api/xero/quotes` against the live connection ‚Üí `403
  insufficient_scope`, with the friendly "Update Xero permissions‚Ä¶" message,
  not a raw SDK error.

Also verified before a real connection existed: `GET /api/xero/status` when
disconnected, `GET /api/xero/connect` with missing credentials (clean
`config_error` naming the exact variable), `GET
/api/xero/quotes|contacts|items` with no connection (`409 not_connected`),
disabled buttons while disconnected, preview panels showing "Connect to Xero
to see ‚Ä¶" instead of an indefinite loading state (a bug caught and fixed
during verification), and responsive layout at 1280√ó900 and 375√ó812.

**Three real issues were found and fixed via this live testing** (not caught
by `tsc`/`eslint`/`build`, since none of them surface without talking to the
actual Xero API):

1. **OAuth callback always failed with `checks.state argument is missing`.**
   `exchangeCodeForTokens` built a fresh `XeroClient` for the token exchange
   without passing the `state` value ‚Äî xero-node's `apiCallback()` validates
   the returned `state` against `this.config.state` internally, so a client
   built without it always throws. Fixed by threading the cookie's expected
   state through into the exchange client (`src/lib/xero/client.ts`,
   `src/app/api/xero/callback/route.ts`).
2. **Every real Xero API error (401/403/etc.) fell through to a generic
   "Unexpected error talking to Xero." / 502**, instead of being classified
   as `insufficient_scope`/`expired_session`. Root cause: xero-node's
   generated Accounting API client rejects with a **JSON-stringified**
   error (not a real `Error`/object), and uses `response.statusCode` rather
   than the `response.status` a standard axios error would have. Fixed
   `toXeroAppError()` in `src/lib/xero/errors.ts` to parse a string payload
   defensively and check both field names ‚Äî while being careful never to let
   the raw string (which includes the live bearer access token in
   `request.headers.authorization`) reach a client-facing message.
3. **Wrong scope requested for Quotes.** The app requested
   `accounting.transactions`, which isn't a plan-tier limitation as first
   suspected ‚Äî it's a scope Xero deprecated for apps created after March
   2026 (this app is one of them, so it's absent from Configuration >
   Authorisation > Scopes entirely, causing an `invalid_scope` error at the
   Xero consent screen). Xero's replacement granular scope for Quotes
   (alongside purchase orders, credit notes, and items) is
   `accounting.invoices`, confirmed via Xero's own developer docs and
   already present in this app's available scopes. Fixed `XERO_SCOPES` in
   `.env`/`.env.example` and the scope check in
   `src/app/api/xero/test-connection/route.ts`.

## Known limitations

- Single-user, single-tenant only (by design, per spec).
- OAuth CSRF state lives in a cookie, not a server-side session table ‚Äî fine
  for one browser/one user, not for multi-device flows.
- "Draft quote writable" is a scope check, not a real write probe ‚Äî this is
  intentional (the spec forbids auto-creating a quote for this check), but it
  means a scope can be granted and the actual write can still fail for other
  reasons (e.g. an org-level restriction).
- SQLite + a locally-generated encryption key are dev-only; both are called
  out in `SETUP_XERO.md` and this report as things to replace before any real
  deployment.
- Contacts/items lists in the preview panels are capped to the first 8 rows
  client-side for a compact dashboard; there's no pagination UI yet.
